# Security Model

This project is a single-site CMS-like app with an admin surface. The security model is intentionally small, but it still needs explicit boundaries.

## Current protections

- admin sessions are stored in hashed form
- session cookies are HTTP-only
- login is rate limited per IP/email combination
- media uploads enforce file type and file size limits
- destructive actions require confirmation in the UI
- public HTML is sanitized before rendering

## HTML safety

The public post renderer does not trust raw HTML directly.

- risky tags are stripped
- event handler attributes are removed
- unsafe URL schemes are removed from links and images
- inline styles are reduced to the safe subset used by the editor

This is especially important for imported WordPress content.

## Admin surface

The admin is protected by session cookies and server-side checks.

- routes use `requireAdmin()` on the server
- API routes that mutate content check authentication server-side
- logout revokes the current session server-side

## Upload safety

Uploads are checked before being stored.

- only allowed image MIME types are accepted
- size is capped at the configured upload limit
- media keys are generated by the server
- the public URL is derived from the storage key, not user input

## Deployment assumptions

- the app is usually behind a trusted proxy/CDN
- `cf-connecting-ip` or trusted forwarded headers may be available
- session cookies should be served over HTTPS in production

## What to verify in a deployment

- admin cookies are not exposed to client JavaScript
- the proxy terminates TLS
- the proxy strips or rewrites client-supplied forwarding headers
- the storage bucket permissions only allow the app to read/write what it needs
- the database user has the minimum required privileges

## Practical hardening checklist

1. Keep the app behind HTTPS.
2. Keep the storage bucket private unless public reads are intentional.
3. Keep uploaded HTML sanitized even if the editor already constrains input.
4. Keep the login rate limiter enabled.
5. Review any new `dangerouslySetInnerHTML` usage before merging.

## Important limitation

The current rate limiter is intentionally simple and in-memory. It is good enough for a single-instance deployment, but a clustered deployment should replace it with a shared store.
